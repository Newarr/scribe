Code reviewer findings

P1

Stale Stop now can still stop recording

AppDelegate.handleEndGuardPrompt reads the prompt generation, but the button closures do not capture it. stopRecordingFromEndPrompt() uses activeEndPromptGeneration at click time and calls stopRecording() unconditionally.

Relevant code: TranscriberApp/Scribe/AppDelegate.swift:1639-1654 and :1689-1694.

let generation = await endGuard.promptGeneration
activeEndPromptGeneration = generation
...
onStopNow: { await self?.stopRecordingFromEndPrompt() }
...
await endGuard.stopNow(generation: activeEndPromptGeneration)
await stopRecording()

Risk: if the prompt was canceled, raced, or replaced, EndGuard.stopNow can no op while AppDelegate still stops the recording.

Suggested fix: capture generation in the HUD closures. Make EndGuard.keepRecording and EndGuard.stopNow return Bool, then only stop or cancel if the generation was accepted.

P1

Stop prompt changes live capture status to stopping too early

While the 10 second HUD is open, capture is still live, but handleEndGuardPrompt sets status = .stopping. The start detection handler only queues new candidates for .recording and .starting.

Relevant code: AppDelegate.swift:1171-1175 and :1641.

if status == .recording || status == .starting {
    queueDetectionCandidate(candidate, event: event)
    return
}
...
status = .stopping

Risk: a new detection candidate during the countdown can present a start prompt while the current recording is still active.

Suggested fix: keep status as .recording while the HUD is counting down, and add explicit end prompt state. Alternatively, include .stopping in the active capture queue path when session != nil.

P2

Ended candidate matching is too broad

isEndedCandidateForCurrentRecording accepts any candidate with the same bundle ID as the current recording.

Relevant code: AppDelegate.swift:1266-1269.

if currentRecordingAppBundleID == candidate.app.bundleID { return true }
return currentRecordingTriggerIdentity == candidate.triggerIdentity

Risk: a queued, stale, or different meeting in the same app can trigger a stop prompt for the current recording.

Suggested fix: prefer trigger identity. Use bundle fallback only for a deliberately modeled identity migration case, such as calendar identity to app identity after calendar expiry.

P2

Silence cancel does not suppress re prompt for 60 seconds

The spec says audio resume during grace or countdown cancels the stop flow and suppresses re prompt for at least 60 seconds. Current EndGuard returns directly to .watching.

Relevant code: EndGuard.swift:214-231.

Risk: a long quiet meeting can flap between HUD and cancellation.

Suggested fix: introduce a short cancel suppression state or timestamp. During that period, observe audio normally but do not fire a new silence based prompt.

P2

AppDelegate behavior tests are too brittle and indirect

The core EndGuard unit tests are useful, but AppDelegate tests mostly inspect source text. They would not catch stale generation clicks or the .stopping start prompt race.

Suggested fix: extract a small end prompt coordinator seam. Add tests with fake EndGuard and fake HUD that assert generation capture, status transitions, and queued candidate behavior.

UX reviewer findings

P1

HUD may not appear over fullscreen meetings or other Spaces

The HUD uses a floating nonactivating NSPanel, but does not set collection behavior such as .canJoinAllSpaces and .fullScreenAuxiliary.

Relevant code: EndCountdownWindow.swift:43-61.

Risk: the stop prompt can be invisible during fullscreen Zoom, Google Meet, or Teams.

Suggested fix: set suitable collectionBehavior and verify in fullscreen Spaces with active meeting apps.

P1

Destructive action is styled as primary and default

The spec says the primary action is Keep Recording. Current UI makes Stop now the primary styled button and Return default.

Relevant code: EndCountdownWindow.swift:127-139.

Risk: users can accidentally stop recording by pressing Return or by following visual emphasis.

Suggested fix: make Keep recording primary and default. Make Stop now secondary. Keep Escape mapped to keep recording or cancel stop.

P1

Copy says Stopping too early

The HUD eyebrow says Stopping · call ended before the user has accepted the stop or the countdown has elapsed.

Relevant code: EndCountdownWindow.swift:76-83.

Risk: users may think recording already stopped, when the app is still recording and waiting for input.

Suggested fix: use Call seems over as the main message, with supporting copy like Scribe will stop in 10 seconds unless you keep recording.

P1

Menu fallback is misleading while countdown is active

While the countdown HUD is active, the menu popover text for .stopping says the app is saving the recording and disables the action. Capture is still live and the user can still keep recording from the HUD.

Relevant code: RecordingMenu.swift:669-719.

Risk: if the HUD is hidden, the menu bar backup gives the wrong state and no recovery action.

Suggested fix: add a distinct Stopping soon or Stop prompt active state with countdown and both actions.

P2

No redundant notification channel

The start prompt has redundant channels. The stop prompt spec also calls for a notification in parallel, especially for users who walked away from the screen.

Suggested fix: add a notification with Keep recording and Stop now actions. Ensure action routing uses the same generation guard as the HUD.

Checklist for follow up agent

Acceptance checklist

Correctness

  • End detection for the active recording opens exactly one stop prompt.
  • Keep Recording snoozes both call ended and silence based prompts for 15 minutes.
  • Stale HUD or notification actions cannot stop the current recording.
  • New start candidates are queued while the stop prompt is open.
  • Silence resume cancels the stop flow and suppresses re prompt for at least 60 seconds.

UX

  • HUD is visible over fullscreen meeting apps and across Spaces.
  • HUD does not steal focus.
  • Keep Recording is the visually primary and default safe action.
  • Menu bar gives a truthful fallback during countdown.
  • Notification actions match HUD actions and respect generation guards.
